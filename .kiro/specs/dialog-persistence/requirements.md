# Requirements Document

## Introduction

Система персистентности диалогов для AI Study Agent. Позволяет сохранять контекст разговоров между сессиями, помнить предыдущие диалоги с каждым персонажем (Остроух, Фейнман, Сократ), восстанавливать контекст при возвращении пользователя и накапливать знания о предпочтениях студента.

## Glossary

- **Checkpointer**: Компонент для сохранения и восстановления состояния диалога между сессиями
- **Conversation_Session**: Отдельная сессия диалога с персонажем, имеющая уникальный идентификатор
- **Dialog_Context**: Контекст диалога, включающий историю сообщений, метаданные и состояние
- **Character**: AI-персонаж (Остроух, Фейнман, Сократ) с уникальным стилем общения
- **User_Preferences**: Накопленные знания о предпочтениях студента (стиль объяснений, темп, интересы)
- **Memory_Window**: Скользящее окно последних N сообщений для контекста LLM
- **Summary**: Сжатое представление длинной истории диалога

## Requirements

### Requirement 1: Сохранение истории диалогов

**User Story:** As a студент, I want чтобы система помнила мои предыдущие разговоры с персонажами, so that я могу продолжить обучение с того места, где остановился.

#### Acceptance Criteria

1. WHEN пользователь отправляет сообщение в чат, THE Checkpointer SHALL сохранить сообщение в базу данных с привязкой к пользователю, персонажу и сессии
2. WHEN пользователь возвращается к диалогу с персонажем, THE System SHALL загрузить последние N сообщений из истории (по умолчанию 20)
3. WHEN история диалога превышает Memory_Window, THE Checkpointer SHALL создать Summary предыдущих сообщений для сохранения контекста
4. THE System SHALL хранить историю диалогов минимум 90 дней

### Requirement 2: Управление сессиями диалогов

**User Story:** As a студент, I want иметь возможность начать новый диалог или продолжить предыдущий, so that я могу организовать свои разговоры по темам.

#### Acceptance Criteria

1. WHEN пользователь открывает чат с персонажем, THE System SHALL показать список предыдущих сессий с этим персонажем
2. WHEN пользователь выбирает "Новый диалог", THE System SHALL создать новую Conversation_Session
3. WHEN пользователь выбирает существующую сессию, THE System SHALL восстановить Dialog_Context этой сессии
4. THE System SHALL автоматически присваивать название сессии на основе первого сообщения или темы

### Requirement 3: Контекст для персонажей

**User Story:** As a AI-персонаж, I want иметь доступ к истории общения со студентом, so that я могу давать персонализированные ответы.

#### Acceptance Criteria

1. WHEN персонаж формирует ответ, THE System SHALL включить в промпт последние сообщения из Memory_Window
2. WHEN история длинная, THE System SHALL включить Summary предыдущего контекста перед Memory_Window
3. WHEN студент упоминает "как мы обсуждали ранее", THE System SHALL найти релевантные сообщения из истории
4. THE System SHALL передавать персонажу информацию о текущей теме обучения студента

### Requirement 4: Накопление предпочтений студента

**User Story:** As a студент, I want чтобы система запоминала мои предпочтения в обучении, so that персонажи адаптировались под мой стиль.

#### Acceptance Criteria

1. WHEN студент явно указывает предпочтение (например, "объясни проще"), THE System SHALL сохранить это в User_Preferences
2. WHEN студент часто задаёт вопросы определённого типа, THE System SHALL обновить профиль предпочтений
3. WHEN персонаж формирует ответ, THE System SHALL учитывать User_Preferences в промпте
4. THE User_Preferences SHALL включать: предпочитаемый уровень детализации, любимые темы, стиль объяснений

### Requirement 5: API для работы с историей

**User Story:** As a разработчик, I want иметь API для работы с историей диалогов, so that я могу интегрировать персистентность в существующий чат.

#### Acceptance Criteria

1. THE API SHALL предоставить endpoint для получения списка сессий пользователя с персонажем
2. THE API SHALL предоставить endpoint для получения сообщений конкретной сессии с пагинацией
3. THE API SHALL предоставить endpoint для создания новой сессии
4. THE API SHALL предоставить endpoint для удаления сессии
5. WHEN сообщение сохраняется, THE API SHALL возвращать ID сохранённого сообщения

### Requirement 6: Производительность и масштабируемость

**User Story:** As a система, I want эффективно работать с большими объёмами истории, so that пользователи не испытывают задержек.

#### Acceptance Criteria

1. WHEN загружается история диалога, THE System SHALL выполнить запрос за время менее 200ms
2. WHEN создаётся Summary, THE System SHALL делать это асинхронно, не блокируя ответ пользователю
3. THE System SHALL использовать индексы для быстрого поиска по userId, characterId и sessionId
4. WHEN история сессии превышает 1000 сообщений, THE System SHALL автоматически архивировать старые сообщения
